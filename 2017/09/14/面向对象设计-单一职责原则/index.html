面向对象设计-单一职责原则

  • java
  • 设计模式

单一职责原则被列在面向对象设计之首，当从字面意思来理解视乎很简单，但是何为单一，何为职责，如何把握职责的粒度，这是该原则不好把控的地方，也是带来争议的点。究其原因，每个人对项目的整体把握度不一致，而被认为是同一个职责的类或者方法在其他项目却不尽相同。

何为单一，何为职责

职责就是Action，单一就是有且只有一个因素可以改变这个职责或者功能。也就是说一个类或者一个方法只能被一种因素使用，即不会有其他的需求导致这个职责的修改或扩展。

定义：单一职责原则就是指一个类或者模块应该有且只有一个改变的原因。

为何要如此

单一职责规定我们将项目分拆成单一的职责单元，主要的目的是为了代码的复用，实现高内聚低耦合。如果一个类或者一个模块的职责过于复杂，那么这样的类或者模块不大会被高性能的复用，一旦类或者模块中的职责多且相互耦合的话，那么对于这个类或者模块来说就会面临的多次被修改的命运。对于代码的维护也带来的巨大的挑战。

上面的接口根据我们正常的角度可以将它分为2个职责，一个是连接操作，一个是通信操作。

那么问题来了，这样的2个职责应该被分开吗？

那就要看项目对于这“两个”职责如何看待了。如果这“两个”职责对于该项目来说只有一个改变因素那就不应该被分开，不然会徒然增加代码的复杂度，过度的设计，如果项目的角度看待的是两个职责，那么对于这个接口来说会有大于一个的因素要求其改变。这样就违反了单一职责原则

那么我们要如何去做呢

为了符合该原则，我们需要将这个接口拆分。

如上面的类图我们将之前的Modem接口拆分成IModemManager和IMessageManager的两个接口。这样对于数据连接部分的职责就会有一个唯一的类来管理，接收消息也有专门的类来管理，降低耦合度。

有问题

那么对于这样粒度的接口定义是否就是完全正确的呢？我们是否真正的做到了单一职责了呢。

这个问题需要交给我们自己对当前项目的思考了。

标准就是这个类是否还存在其他被改变的因素。

同样的如果你在当前项目中已经不存在其他因素来改变数据连接部分了。也就是接口IModemManager的两个方法都遵从同一个因素改变，那么恭喜你这个就是完美的当单一职责类。那么是否就可以把这个接口使用要其他项目中了呢？

答案当然是NO

如果在另一个项目中存在多种拨号方式呢，这样的接口就无法满足单一职责了。所以这一原则是相对的。需要立足在当前项目之上。使用到别处，意义可能就不一样了。

总结

单一职责原则很好理解，难点就是如何去脉搏到项目职责的粒度。这个很关键，将会影响到你对这个类的职责的规划。
