DiskLruCache Java实现基于LRU的磁盘缓存

2016-06-15

DiskLruCache 是基于Java实现的硬盘缓存策略库。使用该库可以将数据，图片等保存在本地，避免了频繁请求服务端的困扰，而且可以极大的提高数据显示的效率。DiskLruCache也被很多的成功的应用使用，如网易新闻，Dropbox和Twitter，而且很多的图片加载库也对缓存策略开放接口，如很有名的Volley，universalimageloader等，我们可以很方便的集成进去。

Github地址 https://github.com/JakeWharton/DiskLruCache

DiskLruCache还是比较简单的，总计也只有3个类。我们来分析下源码。

Util.java

我们先从简单的工具类开始，这个类主要是提供工具方法，

  • readFully(Reader reader)：String

这个方法是将Reader中的数据转换成String，现将Reader中的数据读入Buffer数组中，然后写入StringWriter，调用toString方法，返回数据。

  • deleteContents(File): void

删除文件夹中的所有文件

  • closeQuietly(Closeable): void

关闭Closeable接口。

StrictLineReader.java

用来读取特殊的一行数据

  • fillBuf(): void

in.read(buf, 0, buf.length)正常的读取一个buf.length长度的数据，但是并不能保证读到的数据就是该长度的数据，有可能只读到了1byte的数据。

  • readLine(): String

一开始pos == end == 0,通过判断调用fillBuf() 读取数据，pos =0; end = 读到的数据长度。接着查找读到的数据中是否有换行符，如果有，就表示我们读到了至少是一个完整的一行（可能会多余一行的数据），那么我们截取一行数据返回，第二次调到readLine(),此时的pos = 第二行的起始位置，在判断pos和end如果第一次读到了第二行的数据 就不在调用fillBuff(),接下来继续对读到的数据遍历查找第二行的回车键，如果存在就返回，不存在就创建一个内容大小是输出流原来读到数据长度+80。用来写入第二行数据。接下来进入死循环，首先用创建的输出流接受第一次读到的出去第一行的数据（也就是不完整的第二行数据），调用fillBuff().读取一次数据，判断是否有回车符(即第二行是否完整),如果存在，就将剩下的第二行数据也写入输出流中，返回，如果没有，继续写入，再次读取数据。

  • open(File,int,int,long): DiskLruCache

这方法是库的初始化方法，参数：文件保存路径，app版本，可以保存几分相同文件，最大内存。如果存在备份文件，就用备份文件代替日志文件。第二步创建DiskLruCache实例。第三步读取日志文件。

通过StrictLineReader读取日志的每一行文件。

  • readJournalLine(String): void

lruEntries集合根据日志中的状态放入或者删除一个Entry实例。状态有3种，remove表示移除，相应的也要在集合里移除，clean，dirty，read。都表示数据存在可以获取。

  • processJournal(): void

清楚不正确的日志，比如只有clean日志在的Entry。

  • rebuildJournal(): void

创建新的日志文件，根据lruEntries集合里的Entry。

PendingIntent

在上面多次提到了PendingIntent,它于Intent是有区别的，PendingIntent主要是用于待定的意图，该意图将会在未来的某个时刻发生，而Intent一旦被Start后就会立刻生效，这就是主要的区别。

PendingIntent可以实现三中意图的发生，分别是Activity，Service，广播，分别对应的三个API是

PendingIntent.getActivity(Context context, int requestCode, Intent, int flags);

对应startActivity();

PengdingIntent.getService(Context context, int requestCode, Intent, int flags)

对应startService();

PengdingIntent.getBroadcast(Context context, int requestCode, Intent, int flags)

对应sendBroadcast();

对于第一和第三个参数还是很明白的，主要是第二和第四个参数需要说明。

requestCode表是返回码，于Flag一起标示一个PendingIntent。

一般requestCode设置为0，使用Flag去区别PendingIntent。

Flag有四个常见类型。

  • FLAG_ONE_SHOT: 使用该Flag标记PendingIntent时，标示该意图只会发生一次，之后将不会再次触发意图，在Notification上的表现为，当有多个消息通知栏出现，点击其中的一个触发了意图，之后的消息通知点击后无效。

  • FLAG_NO_CREATE: 使用该Flag设置的PendingIntent时，不会主动创建，在Notification上的表现为。如果没有触发过这个意图的PendingIntent存在，那么点击后无反应，在使用中很少会用到

  • FLAG_CANCEL_CURRENT: 使用这个表示的PendingIntent时，对于已经存在的PendingIntent来说，如果有产生一个PendingIntent的话，之前的都会被取消掉，新创建一个PendingIntent。在Notification上的表现为，之前展示消息通知无法点开，最后一个可以点开。

  • FLAG_UPDATE_CURRENT: 使用这个表示的PendingIntent时，对于之前的PendingIntent都会被更新成最新的Intent，Extras也会被替换。在Notification上的表现为，之前展示的通知与最后一个一样。

上面讨论的是基于同一个PendingIntent，那么匹配原则是：requestCode相同，Intent相同。
requestCode相同很容易理解，Intent相同的原则是：ComponentName和intent-filtre相同，Intent的Extra是不参与匹配的。

Android-IPC通讯Binder和Service的关系

2016-01-05

任何系统都有自己的跨进程通讯，Android是基于Linux，所以都会兼容Linux的通讯机制，包括有管道，信号，socket等等。基于Android自身的特点，有提出了自己的通讯方式Binder机制。接下来就来具体的介绍。

Binder机制

Android中系统的服务基本都是基于Binder，我们在应用层使用的getService()方法调用系统服务。那么这一切是如何进行的呢。

Binder是一种架构，涉及到客户端，服务端，Binder驱动。系统有一个Binder类它将这三个模块连接在一起。Android系统已经帮我们实现了Binder驱动，ServiceManager和系统相关的Service。对于我们开发者来说，我们需要自己去实现需要的Service和Client就可以进行IPC通讯了。

一个Binder服务端就相当于一个Binder对象，一旦被创建，就会在开启一个隐藏线程用来接受client的请求，接受到消息之后就会回调onTransact方法。onTransact方法会根据条件实现不同的服务，那么要使用一个Binder服务就必须要重写onTransact方法。

既然服务端已经具备了接受消息的能力，那么服务端就可以调用transact方法发送消息给服务端，这两个方法是相对应的，我们看下这两个方法的API.

从方法参数中我们就可以知道对应关系。code表示要调用的方法的id,data是调用方法的参数，reply是方法的返回。flags为标记，一般为0.

在Binder驱动中，任意一个服务端Binder被创建后，都在会在驱动中也会创建一个mRemote对象，该对象也是类型也是Binder类。要使用服务端的服务都是通过mRemote来完成的。

在客户端中要使用服务端服务，我们就要获取mRemote的引用，调用transact来发送给服务端消息，服务端接受到我们需要它完成的任务就会运作起来。

在这样的架构里，我们产生了2个Binder对象，一个在服务端，一个在Binder驱动中。对于远程调用来说，我们都是直接调用的是Binder驱动中的mRemote对象。该对象在去调用服务端的Binder对象。

Activity的进阶

2015-12-28

Activity的生命周期变化有两种情况，一个是人为，一个是系统干预。

正常Activity生命周期

对于正常的Activity我们应该很熟悉了。一个Activity从一个不可见状态再变为可见状态有多一个onRestart()回调,接着就进行常规的顺序调用生命周期函数。onStart>onResume,上面说的不可见状态是指在后台。按Home键后，调用周期函数为：onPause>onStop,但是如果是一个透明的Activity，onStop不会被调用。

Activity的周期函数直接有对应关系。onResume和onPause是针对可见性的一组调用，onStart和onStop是针对前后台的一组回调。

当一个Activity A启动Activity B时，A的生命周期会先调用onPause,在去调用B的onCreate>onStart>onResume,等到B展现在前台后，再回去调用A的onStop函数。

非正常Activity生命周期

非正常Activity的生命周期是指系统未经过我们意愿而对Activity的采取的行为。比如系统资源紧张而杀死后台Activity，或者是屏幕切换是的Activity的销毁和创建。

在系统没经过我们同意就去杀死我们Activity，那么也会给我们些补偿措施。onSaveInstanceState()方法会在系统结束Activity时调用的方法，这个方法只会在Activity异常退出时才会调用，执行在onStop之前，与onPause没有时序关系，可前可后。在我们开启一个被系统强制关闭的Activity是，系统会调用onRstoryInstanceState(),在onStart后调用。

这种补偿措施可以保持之前Activity的数据或状态，当我们再次开启时，可以给我们提供还原的场景和机会。

对于系统在Kill Activity时是按照优先级来处理的。前台 > 可见非前台 > 后台。优先kill低的。

对于系统屏幕切换造成的Activity的销毁和重建，我们可以加以控制的。通过配置Activity的android:configChanges我们可以监听感兴趣的属性变化。然后我们在Activity中重写回调函数onConfigurationChanged().这时候系统不会销毁Activity，而是调用该回调方法。

注意的是在4.2后系统默认不会处理screenSize的属性变化了。所以在监听的时候要加上这个属性变化的监听，不然是不会调用onConfigurationChanged()方法的。

Activity的启动模式

我们的Activity都是有所属Task去管理的。启动一个Activity一般会将实例加入Task栈,当销毁时，会出Task栈。Task默认的名字为我么项目的包名，当要启动一个Activity的时候也会默认放入当前context所在的Task,这个就是为什么在使用ApplicationContext或在Service里面启动一个Activity的时候需要加入Flag，这是因为ApplicationContext和Service是没有Task栈的。一个App可以有多个栈，一个栈可以有多个App的Activity.

我们可以在Maninfest中配置Activity的Task。

standard模式

这也是Activity的默认模式，在这种模式下，Activity会新创建一个实例放入当前的Task栈内，没有多余的操作，不管当前Task内存放了怎样的Activity，怎样的顺序，系统都会new 一个实例。

singleTop模式

这种模式会对一个栈内的Activity顺序和个数都没有影响，影响的是设置该模式的Activity，如果启动的Activity在栈内已经有实例了，并且该实例在栈顶，那么就不会重新new Activity，而是使用栈顶的Activity，生命周期也会不会调用onCreate，既然都不是新创建的肯定不会走onCreate回调了，这很好理解。那么就会调用onNewIntent回调。

比如栈内有Activity A B C D,这时候D要启动A,因为栈顶是D所以会按照正常的new A放入Task内，如果D再启动D，栈顶已经是D的实例了，那么就不会new,而是复用改实例，并调用回调函数onNewIntent。

singleTask模式

这种模式会对一个栈内的Activity顺序和个数都有影响。看名字我们就知道在一个Task只允许存在一个实例。很霸道模式，如果一个Task内存在了该Activity的实例，那么Task会将它上面的所有的Activity全部弹出，并且不会调用onCreate而是调用onNewIntent。

比如栈内有Activity A B C D，这时候D要启动A，因为Task已经存在实例A了，那么BCD将会被弹出栈，最后Task里面只剩下A.

这种模式必须配合android:taskAffinity使用，这个配合使用会，被标记的Activity将会作用在指定的Task内，上面没有说是因为android:taskAffinity默认的是项目的包名，也就是在正常启动Activity的Task内，如果要指定作用Task,就必须显示的使用android:taskAffinity。Task名必须要有.。

singleInstance模式

很霸道的模式，不管当前的Task是怎样的顺序，都会新建一个Task,并存放将要启动的activity，如果指定了android:taskAffinity，那么就会会命名新Task的名字，不然新建的名字依然是包名，这样就会出现两个相同名字的Task,但是id不一样，所以id才是Task唯一标示。如果在标记singleInstance的Activity在开启一个Activity，那么这么Activity又会回到原来的Task内，而不是和它在一起，如果我们启动一个singleTask，android:taskAffinity和用singleInstance Activity的android:taskAffinity标记一样的Task的Activity，那么这两个Activity依然不在同个Task里，即使这两个Task同名，而id却不同。

(1)Intent.FLAG_ACTIVITY_NEW_TASK：使用一个新的task来启动Activity，一般用在service中启动Activity的场景，因为service中并不存在Activity栈。
(2)Intent.FLAG_ACTIVITY_SINGLE_TOP：类似andoid:launchMode=”singleTop”
(3)Intent.FLAG_ACTIVITY_CLEAR_TOP：类似andoid:launchMode=”singleTask”
(4)Intent.FLAG_ACTIVITY_NO_HISTORY：使用这种模式启动Activity，当该Activity启动其他Activity后，该Activity就消失了，不会保留在task栈中。例如A B，在B中以这种模式启动C，C再启动D，则当前的task栈变成A B D。

Volley 网络通讯框架一之发起请求

2015-12-11

Android系统给我们提供了一些最直接的HTTP库，和相关的工具类，AsyncTaskLoader，HttpURLConnection，AsyncTask，HTTPClient（Apache）等等，这些东西在使用的时候都没有系统性的管理者，导致在一个APP中很难有效的去管理请求，而且我们也会对除请求之外的事情也要关心，基于此，我们可能需要一个框架来统一我们的网络请求。

所以网络框架的出现也就顺理成章了，接下来就给大家介绍一下Googol出品的volley框架。

发布时间是2013年的IO大会，并且配上了下面的一张宣传图

图片里是发射火弓箭的图，有点类似流星，这里也是表明了volley框架的特点

  1. JSON，图像等的异步下载
  2. 网络请求的排序
  3. 网络请求的优先级处理
  4. 缓存
  5. 多级别取消请求
  6. 和Activity和生命周期的联动（Activity结束时同时取消所有网络请求）

本篇我们会从源码的角度去分析volley网络框架的请求发起的过程

其实是要创建一个RequestQueue对象，而要new 这个对象是需要2个参数的一个是继承了Cache的硬盘缓存对象，一个是继承了Network的网络处理对象，这个我们后面在详细说明，接下来我们要说的是DiskBasedCache和RequestQueue；

RequestQueue.java

如果没有源码可以看着里Guthub For RequestQueue.java

我们看到在Volley中创建了RequestQueue，并且调用了 start() 方法，其实该方法实现的目的就和方法名一样，首先调用stop方法来确保Dispatcher已经是停止的，然后开启我们处理request的两个Dispatcher。这两个Dispatcher我们后面在说。
我们再来看下这里的另一个重要的方法 add(Request request) ;既然启动了Dispatcher，那么我们当然要加入要执行的request了。
它主要工作是：

  1. 将request加入到 HashSet> mCurrentRequests 用来存放正在要处理的request
  2. 判断request是否需要被缓存，如果不是就直接放入 PriorityBlockingQueue> mNetworkQueue 中，而这个队列就是两个Dispatcher其中的NetworkDispatcher要take的队列
  3. 这一步是个优化点：根据request拿到CacheKey 也就是我们以后缓存的句柄，如果 Map>> mWaitingRequests 包含该句柄说明在mWaitingRequests里面有相同的request请求，那么既然是相同的请求，我们就只要请求到一个request就等于访问了所有的request。

根据add这个方法的具体实现那么肯定得有一个request结束后对mWaitingRequests存放的相同的request的处理，那么我们就来看下 finish(Request request) 这个方法

  1. 移除 HashSet> mCurrentRequests 存放的request
  2. 调用onRequestFinished回调
  3. 如果request是需要缓存的，将mWaitingRequests相同的request请求全部取出再放入 PriorityBlockingQueue> mCacheQueue 缓存队列里，因为已经经过一次成功的request请求了，必定会有缓存，所以这里就全部走缓存队列
